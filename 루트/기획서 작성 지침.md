# 기획서 작성 가이드라인

  

이 가이드라인은 어떤 분야의 기획서든 공통으로 적용할 수 있는 원칙을 정의한다.

Claude(AI)에게 기획서 작성을 지시하거나, 사람이 직접 작성할 때 모두 참고한다.

  

---

  

## 핵심 원칙

  

### 원칙 1: 요청사항을 확실하게 정의하라 — 매번 같은 결과가 나와야 한다

  

기획서의 가장 중요한 역할은 **누가 읽어도, 언제 읽어도 동일한 결과를 만들어내는 것**이다.

  

요구사항이 모호하면 사람마다, 시도마다 다른 결과가 나온다. AI에게 지시할 때는 더욱 심각하다 — 같은 기획서로 열 번 요청하면 열 가지 다른 산출물이 나올 수 있다.

  

**요청사항을 확실하게 정의하는 기준:**

- 두 사람이 독립적으로 읽었을 때 동일한 결론에 도달하는가?

- "이게 맞나?"라는 의문이 생기는 부분이 없는가?

- 엣지 케이스(예외 상황)에 대한 처리가 명시되어 있는가?

  

**나쁜 예:** "사용자가 편하게 쓸 수 있는 UI를 만든다."

**좋은 예:** "신규 사용자가 가입 후 핵심 기능을 3분 안에 사용할 수 있어야 한다. 버튼 레이블은 동사+목적어 형식으로 통일하고, 오류 메시지는 원인과 해결 방법을 함께 표시한다."

  

기획서를 완성하면 스스로 질문하라: **"이 기획서만 보고 작업하면 내가 원하는 것이 나오는가?"** 아니라면 더 정의해야 한다.

  

---

  

### 원칙 2: 의도와 목적이 분명하게 정의되어 있어야 한다

  

기획서의 모든 요소에는 **"왜 이것인가"**가 담겨 있어야 한다.

  

무엇을(What) 만들지는 쉽게 적는다. 그러나 왜(Why) 이 방식인지, 이 기능으로 무엇을 달성하려는지가 빠지면 기획서는 단순한 할 일 목록이 된다. 의도가 없는 기획서는 작업자가 판단해야 할 순간마다 방향을 잃는다.

  

**의도와 목적을 정의하는 세 가지 질문:**

- 이것을 왜 만드는가? (존재 이유)

- 이것이 완성되면 누가 어떻게 달라지는가? (변화의 대상과 방향)

- 이것을 만들지 않으면 어떤 일이 생기는가? (부재의 결과)

  

**나쁜 예:** "알림 기능을 추가한다."

**좋은 예:** "사용자가 중요한 이벤트를 놓쳐 이탈하는 문제를 줄이기 위해 알림 기능을 추가한다. 목적은 재방문율 향상이며, 알림이 과도해지면 역효과가 나므로 사용자가 수신 유형을 직접 설정할 수 있어야 한다."

  

의도가 명확하면 기획서를 읽는 사람이 세부 사항에서 막혔을 때 **스스로 옳은 판단을 내릴 수 있다.**

  

---

  

### 원칙 3: 연관된 시스템 사이에는 의도와 목적이 동일해야 한다

  

하나의 제품은 여러 시스템으로 이루어진다. 각 시스템이 서로 다른 의도를 가지고 있으면, 시스템들이 연결되는 순간 충돌이 일어난다.

  

**점검 방법:**

- 연관된 시스템을 나란히 놓고 각각의 의도와 목적을 한 줄로 적는다.

- 그 문장들이 서로 충돌하거나 다른 방향을 가리키지 않는가?

- 상위 목적(제품 전체의 의도)과 개별 시스템의 의도가 일치하는가?

  

**나쁜 예:**

- 결제 시스템 목적: "이탈 없이 빠르게 결제를 완료시킨다."

- 쿠폰 시스템 목적: "사용자가 최대한 많은 쿠폰을 탐색하도록 유도한다."

→ 두 시스템이 같은 화면에서 충돌한다.

  

**좋은 예:**

- 결제 시스템 목적: "사용자가 의도한 구매를 최소한의 단계로 완료한다."

- 쿠폰 시스템 목적: "결제 흐름을 방해하지 않으면서 혜택을 인지시킨다."

→ 두 시스템이 동일한 상위 의도(마찰 최소화)를 공유한다.

  

기획서를 작성할 때 연관 시스템이 있다면 반드시 해당 시스템의 의도와 목적을 함께 기술하고, 일치 여부를 명시적으로 확인한다.

  

---

  

### 원칙 4: 비슷한 역할의 기능은 하나의 메뉴 또는 액션으로 수행할 수 있어야 한다

  

유사한 역할을 하는 기능이 여러 곳에 분산되면 사용자는 같은 목적을 위해 다른 경로를 학습해야 한다. 이는 혼란을 만들고, 제품의 일관성을 해친다.

  

기획서를 작성할 때 새 기능을 정의하기 전에 반드시 물어야 한다. **"이미 비슷한 역할을 하는 기능이 있는가?"** 있다면 새 기능을 추가하는 것이 아니라 기존 기능을 확장하거나 통합하는 방향을 먼저 검토한다.

  

**판단 기준 — 다음 중 하나라도 해당하면 통합을 검토한다:**

- 두 기능의 사용자 목적이 동일하거나 90% 이상 겹친다.

- 사용자가 두 기능의 차이를 설명하기 어렵다.

- 두 기능이 같은 데이터를 다루거나 같은 결과를 만들어낸다.

  

**나쁜 예:** "파일 삭제" 버튼이 목록 우클릭 메뉴, 상세 화면 툴바, 설정 페이지에 각각 따로 존재한다.

**좋은 예:** 삭제 액션은 어디서 시작하든 동일한 확인 모달 → 동일한 결과로 수렴한다. 진입점은 여러 곳이지만 액션은 하나다.

  

기획서에 기능을 추가할 때마다 **기존 기능 목록과 대조**하여 중복 또는 유사 기능 여부를 명시적으로 확인하는 단계를 포함한다.

## Claude에게 기획서 작성을 요청하는 방법

  

### 최소 입력 (이것 없이는 시작하지 말 것)

  

```

1. 무엇을 만드는가 (제품/서비스 이름 + 한 줄 설명)

2. 누구를 위한 것인가 (구체적인 사용자 유형)

3. 어떤 문제를 해결하는가 (현재 사용자가 겪는 불편)

4. 기획서의 독자는 누구인가 (개발팀? 투자자? 내부 리뷰?)

5. 출력 형식 (게임 GDD? 소프트웨어 기획서? 사업계획서?)

```

  

### 프롬프트 예시

  

```

이 가이드라인의 원칙을 따라 다음 제품의 기획서를 작성해줘.

  

제품: TaskFlow — 1인 개발자용 프로젝트 관리 앱

사용자: 사이드 프로젝트를 혼자 개발하는 1인 개발자

문제: 기존 협업 도구(Jira, Notion)가 너무 복잡하고, 단순 할 일 앱은 개발 흐름 추적이 안 됨

독자: 공동 창업자 (기술 검토 및 MVP 범위 결정용)

형식: 소프트웨어 기획서

포함 필수: Kanban + 뽀모도로 타이머 통합 뷰

제외: 팀 협업 기능 일체 (v1 범위 밖)

```

## 기획서 품질 체크리스트

  

기획서를 완성한 후 아래 항목을 점검한다.

  

- [ ] 이 기획서만 보고 작업하면 원하는 결과가 나오는가? (원칙 1)

- [ ] 각 기능에 "왜 이것인가"가 설명되어 있는가? (원칙 2)

- [ ] 연관 시스템의 의도가 서로 충돌하지 않는가? (원칙 3)

- [ ] 유사 기능이 분산되지 않고 하나의 액션으로 통합되어 있는가? (원칙 4)

- [ ] In Scope와 Out of Scope가 모두 명시되어 있는가?

- [ ] 성공 기준이 측정 가능한 형태로 정의되어 있는가?

- [ ] 미해결 질문이 명시적으로 나열되어 있는가?
